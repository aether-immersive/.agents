# .agents/ — Universal AI Agent Configuration

**v1.0.0**

Single source of truth for AI skills, agent personas, hooks, and MCP servers. Works across Claude Code, Gemini CLI, Codex, Cursor, and Antigravity.

## Quick Start

1. **Copy `.agents/` into your project root** (or clone the repo and delete `.git/`)
2. **Open your AI tool** (Claude Code, Gemini CLI, etc.)
3. **Tell it to read `.agents/SETUP-PROMPT.md`** — this scans your codebase, interviews you about your architecture and preferences, and generates all project-specific configuration
4. **Run the update script** to distribute skills and MCP config to each tool:
   ```bash
   # Linux / macOS
   chmod +x .agents/update.sh && .agents/update.sh

   # Windows (PowerShell)
   .\.agents\update.ps1
   ```

That's it. The setup generates `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, agent personas tuned to your stack, and an initial architecture doc. The update script then copies skills to each tool's expected directory.

After initial setup, use `/setup-init` (now available as a skill) or `/setup-update` to refresh configuration.

## How It Works

```
AGENTS.md              ← Canonical project instructions (generated by SETUP-PROMPT.md)
CLAUDE.md              ← @AGENTS.md + Claude-specific directives
GEMINI.md              ← @./AGENTS.md + Gemini-specific directives
.cursor/rules/*.mdc    ← AGENTS.md content wrapped for Cursor (if requested)
AGENTS.md              ← Read natively by Codex and Antigravity

.agents/
├── skills/            ← Reusable slash-command skills (SKILL.md format)
├── agents/            ← Agent personas organized by category
├── hooks/             ← Hook scripts (Claude Code + Gemini CLI only)
├── docs/
│   ├── shared/        ← Curated team knowledge (tracked in git)
│   └── private/       ← Local working notes (gitignored)
├── mcp.json           ← MCP server definitions
├── update.sh          ← Syncs config to tool-specific directories
└── update.ps1         ← Windows equivalent
```

**Instructions** live in `AGENTS.md`. Tool-specific files (`CLAUDE.md`, `GEMINI.md`) use `@import` to reference it. Codex and Antigravity read `AGENTS.md` natively. Cursor gets a generated `.mdc` file.

**Skills** live in `.agents/skills/`. The update script copies them to `.claude/skills/`, `.gemini/skills/`, etc. Skills use progressive disclosure — only name + description loaded at startup, full content on invocation.

**Agent personas** live in `.agents/agents/` organized by category. Skills spawn them as needed — they're loaded by explicit path, not by discovery.

**Hooks** live in `.agents/hooks/`. Referenced by path in `.claude/settings.json`. Only Claude Code and Gemini CLI support hooks.

**MCP servers** defined in `.agents/mcp.json`. Copied to `.mcp.json` and `.cursor/mcp.json` by the update script.

## Platform Support

| Feature | Claude Code | Gemini CLI | Codex | Cursor | Antigravity |
|---------|-------------|------------|-------|--------|-------------|
| Instructions | `CLAUDE.md` @imports `AGENTS.md` | `GEMINI.md` @imports `AGENTS.md` | Reads `AGENTS.md` natively | `.cursor/rules/*.mdc` | Reads both |
| Skills | `.claude/skills/` | `.gemini/skills/` | `.codex/skills/` | N/A (rules only) | `.antigravity/skills/` |
| Hooks | Yes (enforcement-grade) | Yes (pre/post tool) | Limited | No | Via Gemini |
| MCP Servers | `.mcp.json` | `settings.json` | `config.toml` | `.cursor/mcp.json` | Via Gemini |

The update script handles Claude Code, Gemini CLI, and Cursor. Codex reads `AGENTS.md` natively. For Antigravity, use the Gemini config.

## Adding a Skill

1. Create `.agents/skills/{category}-{name}/SKILL.md`
2. Run `update.sh` (or `update.ps1`)
3. Invoke with `/{category}-{name}` in any supported tool

### SKILL.md Format

```yaml
---
name: category-action-name
description: >
  When to use this skill. Be specific — this is what the agent reads
  to decide whether to auto-invoke it.
---

# Skill Title

Instructions for the agent when the skill is invoked.
```

### Naming Convention

Use `{category}-{action}` with single hyphens. The category prefix groups related skills in autocomplete.

Rules:
- Lowercase, hyphens, numbers only (`a-z`, `0-9`, `-`)
- No consecutive hyphens (`--`)
- Directory name must match the `name` field in SKILL.md
- Keep under 64 characters
- Skills are **flat** (one directory level) — skill discovery is not recursive

## Adding an Agent Persona

Create a markdown file in `.agents/agents/{category}/`:

```
.agents/agents/
├── review/
│   ├── security-auditor.md
│   └── ...
├── research/
│   └── ...
└── your-category/
    └── your-agent.md
```

Agents are organized by category subdirectory. They're loaded by explicit path, not by discovery.

To use an agent:
- Reference it in a skill (like `/review-council` does)
- `@import` it in CLAUDE.md/GEMINI.md for persistent loading (burns context window)

## Adding a Hook

Create a script in `.agents/hooks/`, reference it in `.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": ".agents/hooks/my-hook.sh" }]
    }]
  }
}
```

Hooks only work in **Claude Code** and **Gemini CLI**. Cursor and Codex do not support hooks.

## Knowledge Base

Two-tier system for capturing and sharing knowledge. Private notes are local and gitignored. Shared notes are committed to git for the team.

### Private (`.agents/docs/private/` — gitignored)

Where `/docs-save-note` writes by default. Local to this machine.

```
.agents/docs/private/
├── debugging/       # Bug fixes and root cause analysis
├── architecture/    # Design decisions and patterns
├── tooling/         # Build system, dev environment
├── convention/      # Coding standards and style
├── mistake/         # Errors and lessons learned
├── preference/      # User workflow preferences
└── _archive/        # Stale notes (archived, not deleted)
```

### Shared (`.agents/docs/shared/` — tracked in git)

Curated knowledge promoted from private via `/docs-promote-notes`.

Same directory structure as private. Only stable, validated, team-relevant notes here.

### Flow

```
/docs-save-note  →  private/ (local, gitignored)
                          ↓  /docs-promote-notes
                       shared/ (tracked in git)
```

### Note Format

```yaml
---
tags:
  - ai-generated
  - {category}
date: YYYY-MM-DD
source: claude-code
related: []
---

# {Title}

## Context
{One sentence: what were you doing?}

## Detail
{Specific knowledge. File paths, error messages, code snippets.}

## Resolution
{What fixed it, or what the correct approach is.}

## Related Files
- `path/to/file.ts`
```

## Maintaining the .agents System

Use the built-in skills to keep things consistent:

- `/repo-housekeeper` — Audits `.agents/` structure, naming, sync status
- `/skills-adopt-orphans` — Finds skills created outside `.agents/skills/` and migrates them
- `/research-docs-sync` — Syncs docs with codebase changes
- `/setup-update` — Refreshes config after major stack changes

The `check-skill-sync` hook automatically warns when you create a skill in the wrong location.

## Adopting Orphan Skills

If you create a skill directly in `.claude/skills/` or `.gemini/skills/` (outside canonical `.agents/skills/`), the PostToolUse hook will warn you. Run `/skills-adopt-orphans` to migrate it.
